---
title: Setup and Passwords
description: How first-run setup works, how to skip it, and where generated passwords are stored.
---

## Default Behavior

Deployery shows a one-time `/setup` page the first time an instance starts without a configured `code-server` password.

The setup page:

- asks you to choose the first password
- writes a `hashed-password` into `~/.config/code-server/config.yaml`
- hands off to normal `code-server` login after that

This is the default behavior for all deployment targets.

## Skip the Setup Page

If you want `code-server` to generate the first password instead, set:

```bash
DEPLOYERY_SKIP_SETUP=1
```

`true` should also be accepted.

When this flag is enabled and no password is configured yet:

- Deployery does not block on `/setup`
- `code-server` creates its normal `config.yaml`
- `code-server` generates a random first password

## Where the Generated Password Lives

The generated password is written to:

```text
/deployery/home/user/.config/code-server/config.yaml
```

The relevant section looks like this:

```yaml
auth: password
password: your-generated-password
```

`code-server` logs that it wrote the config file, but it does not log the generated password itself. The config file is the source of truth.

## Retrieve the Generated Password

### Local Docker

```bash
docker exec <container-name> cat /deployery/home/user/.config/code-server/config.yaml
```

### DigitalOcean Droplets

If you are running Docker on the Droplet, you can use:

```bash
docker exec <container-name> cat /deployery/home/user/.config/code-server/config.yaml
```

If `/deployery` is bind-mounted from the host, you can also read the file directly from the mounted host path.

### Render

Open a shell for the running service and read:

```bash
cat /deployery/home/user/.config/code-server/config.yaml
```

### Railway

Open a shell for the running service and read:

```bash
cat /deployery/home/user/.config/code-server/config.yaml
```

## Notes

- The generated password path only matters when `DEPLOYERY_SKIP_SETUP` is enabled.
- After a password is present in `config.yaml`, Deployery should go straight to normal `code-server` login.
- Future password reset should be a separate authenticated flow, not a reuse of `/setup`.
